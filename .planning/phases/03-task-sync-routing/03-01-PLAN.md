    3. In the sync-to-calendar handler (~line 282-286):
       - For create: `const eventId = await createCalendarEvent(token, augmented, defaultPushId ?? undefined)` — when defaultPushId is null the parameter falls through to the 'primary' default per D-02
       - For update (line ~283): `await updateCalendarEvent(token, task.google_event_id, augmented)` — per D-04, do NOT pass defaultPushId for updates; update goes to whichever calendar already has the event (omitting calendarId uses 'primary' default, which matches existing events)
    4. In the delete handler (~line 310): `await deleteCalendarEvent(token, task.google_event_id!)` — same as update, delete from wherever the event lives per D-04

    NOTE on D-04: Updates and deletes target the calendar where the event already exists. Since we don't store which calendar was used at sync time (deferred), we pass no calendarId for update/delete so they hit 'primary'. This is correct for existing events. For newly created events that went to a non-primary calendar, the event ID is unique across calendars, but the API requires the right calendar. This is a known limitation acknowledged in the deferred items — for v1, all updates/deletes go to 'primary' unless the event was created there. 
    
    ACTUALLY — re-reading D-04 more carefully: "update it in whichever calendar already has it." The Google Calendar API requires the correct calendarId to update/delete. Since we don't store which calendar was used, we need to pass defaultPushId for updates/deletes too, with the understanding that if the user changes their default push calendar, old events won't be findable. This matches the "deferred" note about storing calendarId at sync time.

    Revised approach: Pass `defaultPushId ?? undefined` to ALL three calls (create, update, delete). This routes everything through the configured calendar consistently.
